# Проверка данных 
## По структуре
+ **Ненулевые поля**

Когда мы точно знаем, что данные в колонке не могут быть пустыми, а факт пустоты есть ошибка

* **Уникальность**

Колонка или их комбинация либо несколько комбинаций могут быть уникальны по логике и при этом нарушение уникальности может привести к дуплекации при соеденении полученной таблицы с дуругими данными

* **Соответствие перечисляемым значениям (списку)**

Иногда колонка по логике может содержать перечисление, такие как название дней недели или коды валют и тп

* **Соответствие шаблону**

Колонка может содержать слаботипизированное значение
Примером слаботипизированного значения могут быть: 
 + Дата хранимая в строке  в определенном формате
 + число в строке  
 + номер договора состоящий из  комбинации 4х букв, пробела и 6 цифр (^\w(4) \d(6)$)

* **По соответствию ожиданиям бизнеса**

В этом случае формируется запрос к полученным данным который должен определить его соответствие бизнессмыслу или шаблону
такой запрос может содержать обращение к иным источникам хранилища либо оригинальных баз данных

* **Соответствие значениям из внешнего справочника (ссылочная целостность)**
Значения в колонке датасета должны соответствовать значениям в справочнике

## По влиянию на систему 
* **Контролируемые** 
требующие прерывания пайплайна сбора для предотвращения мутации данных в послеидущих датасетах

* **Информативные** 
используются для анализа состояния качества без четких SLA 

## По детализации 

* **Срабатывание в моменте выполнения в виде кодированной ошибки или сообщения в логе**
Такое может быть использовано на в момент сборки данных

* **Сохранение результата проверки в виде статуса**
Сохраненный статус уже может служить основой для несложного анализа

* **Сохранение всех артифактов проверки полностью или на выбор**
В процессе проверки создаются выборки, результаты которых должны содержать указание на найденные аномалии
Эти аномалии представляют аналитическую ценность при глуюлком разборе причин, это ускоряет разбор проблемы

## По способу установки
* **Параметризация исполняющего приложения**
Хорошо подойдет для конструктивов (pk fk uk check)

* **SQL запрос в свободной форме**
 Универсален для всех случаев, особенно полезен для бизнесс проверок

## По моменту проверки
* **В процессе формирования данных**
* **В виде отдельного этапа сбора** 

Таким образом проверка качества должна обладать следующими свойствами:
* Делать запрос как полученному инкременту так и к конечному датасету 
* Иметь возможность связать датасет или инкремент с другими имсточниками
* Иметь возможность сохранить результат проверки, чтобы им можно было воспользоваться
* быть способной встроится как в пайплайн формирования так и в приложения делающие данные
