# Data-engineering-theories
Это просто запись идей, о том как в теории должен выглядеть сервис подготовки данных.
Из каких функциональных частей он должен состоять.

Главная декларируемые цели надежность, доступность и простота обслуживания.

Сейчас opensource предлагает множество решений для конкретных целей. Как правило одни решения не учитывают работу других решений. Чтобы заставить их работать вместе  обычно пишется интеграционный код.
Который начинает хаотично разростаться. почему это происходит?

Как правило причина разростания ростущие аппетиты бизнессзаказчиков(и это нормально). Как только реализуем одну потребность, открывается еще несколько. Как правило разработка несколько  отстает от потребностей бизнеса. Часто мы получаем систему полную различных допущений. Допущения позволяют показать продукт быстрее и это выглядит привлекательным и иногда даже вполне себе оправданно. Но в какойто момент эти допущения перестают давать привлекательный профит в виде быстрой реализации. Накопленные допущения (техдолг) начинают требовать все больше ресурсов для обслуживания.
Тогда наступает момент для пересмотра получившейся системы, выделения вредных допущений и перевод их в что то более системное и самообслуживаемое.

Данная теория предлагает неиспользовать лоскутное одеяло из предлагаемого сообществом множества инструментов, а посмотреть на проблематику подготовки данных в комплексе. 
В этом комплексе есть две атомарные абстракции. Датасет - объект хранящий данные, обладающий различными свойствами
и процесс - некий способ манипулирования как изменениями объекта, так и сбором различной информации об объекте
