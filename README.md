# Data-engineering-theories
Это просто запись идей, о том как в теории должен выглядеть сервис подготовки данных.
Из каких функциональных частей он должен состоять.

Главные декларируемые цели - это надежность, доступность и простота обслуживания.

Сейчас opensource предлагает множество решений для конкретных целей. Как правило, одни решения не учитывают работу других решений. 
Чтобы заставить их работать вместе, пишется интеграционный код.

Этот код начинает хаотично разрастаться. Почему это происходит?

- Как правило, причина растущие аппетиты бизнес заказчиков(и это нормально). Как только реализуем одну потребность, открывается еще несколько. Как правило, разработка несколько отстает от потребностей бизнеса. Часто мы получаем систему полную различных допущений. Допущения позволяют показать продукт быстрее и это выглядит привлекательным и иногда даже вполне себе оправданно. Но в какой-то момент эти допущения перестают давать привлекательный профит в виде быстрой реализации. Накопленные допущения (техдолг) начинают требовать все больше ресурсов для обслуживания.
Тогда наступает момент для пересмотра получившейся системы, выделения вредных допущений и перевод их во что-то более системное и самостоятельно обслуживаемое.

- Сами задачи не всегда удается решать комплексно, тк часто работа ведется в отсутствии команды разработки платформы данных, а ее этой команды возложены на разработчиков бизнес данных.
Задачи решаются в конкуренции с бизнес-задачами. Получают недостаточный ресурс, чтобы выполнить комплексную доработку. 

Данная теория предлагает не использовать лоскутное одеяло из предлагаемого сообществом множества инструментов, а посмотреть на проблематику подготовки данных в комплексе. 
Используя известные и проверенные фреймворки.

В этом комплексе есть две атомарные абстракции. Датасет - объект хранящий данные, обладающий различными свойствами
и процесс - некий способ манипулирования как изменениями объекта, так и сбором различной информации об объекте.
