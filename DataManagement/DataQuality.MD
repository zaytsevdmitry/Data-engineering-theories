# Качество данных

Выражается в соответствии с ожиданиями от данных и реализуются путем проверки соответствия ожиданиям.
## Польза
* **Уверенность**

Пользователи данных уверены в данных с которыми работают

* **Снижение трудозатрат**

В случае сбоев процессов формирования данных инженер должен восстановить процессы и убедится в том, что данные сформированы. Проверку данных должна выполнить система контроля качества данных, быстро, массово и без отрыва от процесса формирования. Настроенный процесс проверки может скомпенсировать недостаток экспертизы инженера в каком либо вопросе, или снизить когнитивную нагрузку на команду. Уменьшит количество рутинных действий.

## Способ реализации 
Реализация абстракции процесс.
Может быть задачей для планировщика или сервисом
## Виды проверок
### По структуре
+ **Ненулевые поля**

Когда мы точно знаем, что данные в колонке не могут быть пустыми, а факт пустоты есть ошибка

* **Уникальность**

Колонка или их комбинация либо несколько комбинаций могут быть уникальны по логике и при этом нарушение уникальности может привести к дупликации при соединении полученной таблицы с другими данными

* **Соответствие перечисляемым значениям (списку)**

Иногда колонка по логике может содержать перечисление, такие как название дней недели или коды валют и тп

* **Соответствие шаблону**

Колонка может содержать слабо типизированное значение
Примером слабо типизированного значения могут быть: 
 + Дата хранимая в строке в определенном формате
 + число в строке  
 + номер договора состоящий из комбинации четырех букв, пробела и 6 цифр (^\w(4) \d(6)$)

* **По соответствию ожиданиям бизнеса**

В этом случае формируется запрос к полученным данным который должен определить его соответствие бизнес-смыслу или шаблону
такой запрос может содержать обращение к иным источникам хранилища либо оригинальных баз данных

* **Соответствие значениям из внешнего справочника (ссылочная целостность)**
Значения в колонке датасета должны соответствовать значениям в справочнике

### По влиянию на систему 
* **Контролируемые** 
требующие прерывания пайплайна сбора для предотвращения мутации данных в следующих датасетах

* **Информативные** 
используются для анализа состояния качества без четких SLA 

### По детализации 

* **Срабатывание в моменте выполнения в виде кодированной ошибки или сообщения в логе**
Такое может быть использовано в момент сборки данных

* **Сохранение результата проверки в виде статуса**
Сохраненный статус уже может служить основой для несложного анализа

* **Сохранение всех артефактов проверки полностью или на выбор**
В процессе проверки создаются выборки, результаты которых должны содержать указание на найденные аномалии
Эти аномалии представляют аналитическую ценность при глубоком разборе причин, это ускоряет разбор проблемы

### По способу установки
* **Параметризация исполняющего приложения**
Хорошо подойдет для конструктивов (pk fk uk check). В исполняющем приложении должны быть заложены шаблонные проверки, которые можно параметризовать 

* **SQL запрос в свободной форме**
 Универсален для всех случаев, особенно полезен для проверок бизнес-логики.
В исполняющем приложении должны быть заложены интерфейсы для добавления запросов и стандартные реакции на результаты выполнения

* **Код на целевом языке программирования**
Способ для сложных случаев, на пример когда используемое хранилище не способно предоставить SQL на пример kafka/mongodb предоставляют собственное api. В исполняющем приложении должны быть заложены интерфейсы для добавления классов/скриптов и т.д. и стандартные реакции на результаты выполнения

### По моменту проверки
* **В процессе формирования данных**
* **В виде отдельного этапа сбора** 

### Варианты использования и реакции 

* **Нотификация указанных потребителей**
Оповещение по почте или в канал месенжера


* **Предотвращение запуска** 

Внедрение механизмов, предотвращающих запуск процессов подготовки данных использующих "плохой" датасет

- Ведение матрицы качества, где по оси Y датасеты, а по оси X временной интервал, пересечение заполняется количеством неуспешных проверок.

## Требования к реализации
Таким образом проверка качества должна обладать следующими свойствами:
* Делать запрос как полученному инкременту так и к конечному датасету 
* Иметь возможность связать датасет или инкремент с другими источниками
* Иметь возможность сохранить результат проверки, чтобы им можно было воспользоваться
* быть способной внедриться как в пайплайн формирования так и в приложения делающие данные
* Обладать широким спектром интерфейсов доступа к данным
