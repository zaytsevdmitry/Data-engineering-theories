# Управление конфигурацией
Выражается в виде сервиса с api и баазы данных с несложным интерфейсом позволяющим извлекать данные и комбинировать их. 

## Польза

* **Катологизация и учет** Объектов данных 

Наличие корневых данных в одном месте позволяет связывать их с любой дополнительной (мета) информацией в зависимости от потребностей организации.

возможными атрибутами данных могут стать : 

-- Владелец(заказчик)

К атрибутам можно привязать метрическую информацию, например : 

-- размер использованного хранения

-- затраты на RAM/CPU (GPU)

а также применяя тарификацию вести учет расходов на хранилище и компьют


* **Единый подход к определению данных** 

Вне зависимости от происхождения источников Объектов данных их ведение обеспечивается в едином виде, что снижает когнитивную нагрузку на всех пользователей и  инженеров хранилища, упрощает разработку и обслуживание данных.

## Возможные подходы 
может включать разные инструменты и подходы
* **Конфигурация как код**

Одновременно является как наиболее гибким так и наиболее запутанным(для конечного потребителя) способом. 
Если следовать парвилам может стать вполне понятным и удобным инструментом с точки зрения инжиниринга и наоборот очень сложно поддерживаемым в неустоявшейся системе, особенно при росте количества датасетов.
чувствителен к появлению новых требований к вторичным функциям ( с точки зрения конфигурирования), таким как предоставление метаданных и мониторинг

* **Документация как конфигурация**

Метод ставит своей целью учет потребностей в документировании на стадии конфигурирования, прямое использование артефактов документировании в моменте выполнения
На пример 
 -- информация о принципе партиционирования используется как в документации так и непосредственно в процессе формирования данных и не может стать неактуальной
 -- состав типы и описания колонок находят свое место как в процессе формирования данных так и в описании в документации
 -- состав зависимостей объекта так же может служить как в моменте выполнения так и для описания data linage


Первый подход может проигрывать второму в системах, где требуется больше удилять внимания данным, чем способам их интеграции.





## Доставка изменения конфигурации. Организация процесса
Способ доставки конфигурации должен диктоваться потребностями организации, прибывать в балансе рисков и допущений. 
* Одним из оптимальных способов является использование git в качестве интерфейса внесения изменений. т.к. он изначально создан для работы с версионированием кода, а такие инструменты как 
github и gerrit позволяют провести допуск к кода к применению путем проведения ревью и выполнением валидаций триггерами githook
После применения изменений в целевую ветку (мастер) конфигурация поступает под контроль автоматической системы. Сервис-хранитель конфигурации должен получить и загрузить конфиг и ожидать запросов. 

Это может выглядеть следующим образом: 

git hook информирует Сервис cvs об изменении в целевой ветке. Сервис cvs загружает ветку себе, считывает и отправляет изменения в Сервис-хранитель конфигурации

## Использование конфигурации
Данный подход имеет единственную ответственность - хранение состояния конфига.
### Применение конфигурации в моменте формирования данных
Ответственность запуска исполняющего приложения лежит на адаптере который должен получить конфигурацию и выполнить в runtime конфигурирование исполняющео движка
в том случае если речь о неоднородном движке 
Это может быть sqoop dbt flink или  spark приложение
Адаптер может быть запущен любым dag sheduller если задача является периодичной или же реализованным в виде сервиса (в k8s) если задача предполагает непрерывный характер работы.  
### Использование конфигурации для документирования
Конфигурация может выгружаться единым процессом в системы документирования, такие как confluence, openmetadata  или же использоваться напрямую из места хранения
что наталкивает на применение в качестве места хранения субд на основе sql (наиболее перспективный выбор postgres)
### Разработка вспомогательных инструментов
Возможно создание инструментов помогающих пользователю конфигурировать данные, которые в том числе могут использовать текущее состояние конфига для формирования файлов изменения конфигурации

## Компоненты
* git - ответственность: хранение истории изменений
* Сервис cvs - передача утвержденной конфигурации в среду выполнения
* Сервис-хранитель конфигурации - rest контроллер принимает конфигурации и раздает их по запросу

## Дополнительные возможности:
* **Аналитика метаданных хранилища** - имеет высокую ценность с точки зрения управления большим хранилищем со множеством объектов (таблицы/датасеты)
* **UI/CLI** наличие сервиса конфигурации открывает возможности для разработки пользовательских интерфейсов
* **Разработка вспомогательных инструментов** Возможно создание инструментов помогающих пользователю конфигурировать данные, которые в том числе могут использовать текущее состояние конфига для формирования файлов изменения конфигурации
