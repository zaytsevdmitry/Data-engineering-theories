# Управление конфигурацией
Выражается в виде сервиса с api и базы данных с несложным интерфейсом позволяющим извлекать данные и комбинировать их. 

## Польза

* **Катологизация и учет** Объектов данных 

Наличие корневых данных в одном месте позволяет связывать их с любой дополнительной (мета) информацией в зависимости от потребностей организации.

Возможными атрибутами данных могут стать: 

- Владелец(заказчик)

К атрибутам можно привязать метрическую информацию, например: 

- размер использованного хранения

- затраты на RAM/CPU (GPU)

А также применяя тарификацию вести учет расходов на хранилище и вычисления


* **Единый подход к определению данных** 

Вне зависимости от происхождения источников Объектов данных их ведение обеспечивается в едином виде, что снижает когнитивную нагрузку на всех пользователей и инженеров хранилища, упрощает разработку и обслуживание данных.

## Возможные подходы 
Может включать разные инструменты и подходы
* **Конфигурация как код**

Одновременно является как наиболее гибким, так и наиболее запутанным(для конечного потребителя) способом. 
Если следовать правилам может стать вполне понятным и удобным инструментом с точки зрения инжиниринга и наоборот очень сложно поддерживаемым в неустоявшейся системе, особенно при росте количества датасетов.
Чувствителен к появлению новых требований к вторичным функциям (с точки зрения конфигурирования), таким как предоставление метаданных и мониторинг

* **Движимый методанными**

Реализация подхода, основанного на метаданных: используя описательные данные (данные о данных) для динамического управления и настройки систем позволяет настраивать ETL/ELT-конвейеры, интеграции данных. Гибкое управление данными — вместо использования жестко закодированной логики. Хранение конфигурации во внешних таблицах или файлах (например, SQL/JSON) обеспечивает масштабируемые, гибкие и автоматизированные процессы.

**Ключевые характеристики и преимущества**
Сокращение трудозатрат на разработку: Вместо переписывания кода для каждого нового источника, один универсальный конвейер может обрабатывать несколько источников, используя метаданные для определения поведения.
Повышенная гибкость: Изменения в структуре данных или правилах преобразования можно обрабатывать, просто обновляя таблицу метаданных, а не изменяя код.
Улучшенное обслуживание: Централизованные метаданные обеспечивают более четкое представление о происхождении данных, контроле доступа и требованиях соответствия.
Масштабируемость: Позволит организациям эффективно управлять огромными объемами данных, разделяя логику обработки данных и логику преобразования.
Автоматизация: Метаданные управляют автоматизированными преобразованиями и запуском событий.

Распространенные сценарии использования
* Управление данными и качество: Метаданные предоставляют контекст о происхождении и структуре данных, что позволяет автоматизировать проверки качества и обеспечивать соответствие требованиям.
* Метаданные используются в качестве «связующего звена» для интеграции различных источников данных и организации обработки.
* Метаданные применяются для документирования на стадии разработки датасетов, прямое использование артефактов конфигурации позволяет динамически поддерживать документацию

На пример 
 - информация о принципе партиционирования используется как в документации так и непосредственно в процессе формирования данных и не может стать неактуальной
 - состав типы и описания колонок находят свое место как в процессе формирования данных так и в описании в документации
 - состав зависимостей объекта так же может служить как в моменте выполнения так и для описания data linage



## Доставка изменения конфигурации. Организация процесса
Способ доставки конфигурации должен диктоваться потребностями организации, прибывать в балансе рисков и допущений. 
* Одним из оптимальных способов является использование git в качестве интерфейса внесения изменений. т.к. он предоставляет возможность версионирования кода.
Такие инструменты как github и gerrit позволяют провести допуск кода к применению путем проведения ревью и выполнением валидаций триггерами githook
После применения изменений в целевую ветку (мастер) конфигурация поступает под контроль автоматической системы. Сервис-хранитель конфигурации должен получить и загрузить конфиг и ожидать запросов. 

Это может выглядеть следующим образом:

Git hook информирует Сервис cvs об изменении в целевой ветке. Сервис cvs загружает ветку себе, считывает и отправляет изменения в Сервис-хранитель конфигурации

## Использование конфигурации

Данный подход имеет единственную ответственность - хранение состояния конфига.

### Применение конфигурации в моменте формирования данных

Ответственность запуска исполняющего приложения лежит на адаптере который должен получить конфигурацию и выполнить в runtime конфигурирование [процесса](AbstractEntities/Process.MD)
в том случае если речь о неоднородном движке 
Это может быть sqoop dbt flink или spark приложение
Адаптер может быть запущен любым dag [планировщике](Scheduler.MD) если задача является периодичной или же реализованным в виде [сервиса](DataService.MD) (в k8s) если задача предполагает непрерывный характер работы.  

### Использование конфигурации для документирования
Конфигурация может выгружаться единым процессом в системы документирования, такие как confluence, openmetadata или же использоваться напрямую из места хранения, что наталкивает на применение в качестве места хранения субд на основе sql (наиболее перспективный выбор postgres)

### Разработка вспомогательных инструментов
Возможно создание инструментов помогающих пользователю конфигурировать данные, которые, в том числе могут использовать текущее состояние конфига для формирования файлов изменения конфигурации

## Компоненты
* git - ответственность: хранение истории изменений
* Сервис cvs - передача утвержденной конфигурации в среду выполнения
* Сервис-хранитель конфигурации - rest контроллер принимает конфигурации и раздает их по запросу

## Дополнительные возможности:
* **Аналитика метаданных хранилища** - имеет высокую ценность с точки зрения управления большим хранилищем со множеством объектов (таблицы/датасеты)
* **UI/CLI** наличие сервиса конфигурации открывает возможности для разработки пользовательских интерфейсов
* **Разработка вспомогательных инструментов** Возможно создание инструментов помогающих пользователю конфигурировать данные, которые, в том числе могут использовать текущее состояние конфига для формирования файлов изменения конфигурации
