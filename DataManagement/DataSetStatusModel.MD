# Статусная модель данных датасетов
Описывает именованным статусом  инкрементальное состояние датасета.
по мимо именования интервал характеризуется  моментом начала и моментом окончания. Эти значения присутствуют в качестве фильтра к источнику данных тк это одни и теже значения.

# Проблематика
## Инкрементальное состояние не всегда равно интервалам запуска планировщика заданий.
На пример есть датасет который мы собираем раз в сутки, но запрос формирующий инкремент охватывает более глубокий период чем дата прошлого запуска.
Как правило такая необходимость вызва на логикой изменения источника. На пример источник может получать изменения записей, которые созданы за пределом интервала а отметка с указанием момента изменения не предоставляется.
## Интервалы запуска планировщика могут меняться.
На пример сначала нам нужно прогрузить историю крупными партиями по 1 году, а потом мы переходим в режим повседневного сбора суточными интервалами 

## Датасеты меняющиеся онлайн и/или в потоке
по таким датасетам сложно отслеживать интервалы инкрементов

* ни одна потоковая система не предоставляет подобной функциональности тк считается, что это элемент логики данных.

* как и в случае с расписанием на стороне отправляющей системы требуется обеспечение датой измениния

иначе говоря отправляющая система обязана обеспечить что то вроде технологического поля с моментом изменения. Принимающая система обязана обработать само сообщение, поле с моментом изменения , собщить в сервис статусную запись. выполняется что то вроде двухфазной фиксации.

для случаев когда отправляется изменение класса crud важен порядок изменений, поэтому требуется еще одно служебное поле счетчик.

в ассинхронной среде нужно обеспечить отправку ключа записи в ключь распределения
это также обязанность отправляющей системы

# Варианты использования 
* В механизме котнтроля в процессе сборки датасета - проверяются зависимость на предмет готовности их использования
* В DataQuality - отображение полноты и свежести датасета 
* В мониторинге - составление трешхолдов и нотификаций

# 
Очевидно инкременты характеризуются интервальностью и статусом
поэтому модель может иметь следующий вид:
```
-------------------
|name| type| incr_from|incr_to|
-----------------
|
```
При этом не нужно учитывать периодичность 

# Контракты

## Изменение состояния 

```json
{
dataResource: "mydataset"
incr_from: "2020-01-01 00:00:00",
incr_to: "2021-01-01 00:00:00",
}
Сервис должен зафиксировать полученный интервал, и отправить уведомления об изменении всем подписчикам.
```
## Запрос состояния

```json
{
dataResource: "mydataset",
incr_from: "2020-04-01 00:00:00",
incr_to: "2020-05-01 00:00:00"
}

```
в ответ на такой запрос сервис производит поиск "осутствующих данных" в указанном временном ограничении.Проверяются состояния не только самого датасета, но и связанных с ним проверок качества данных. Если в запрошенном интервале есть пробелы, то сервис возвращает их списком, чтобы клиент помимо отказа в операции, мог еще и вывести причину отказа, по которой можно начинать принимать меры. В случае успеха сервер отправляет статус, с пустым списком.



пример: нехватает 1 суток

```json
{
status: "FAILED",
intervals: [
  {
   type: dataresource,
   name: "mydataset",
   incr_from: "2020-04-30 00:00:00",
   incr_to: "2020-05-01 00:00:00"
  },
  {
   type: dataqualityMetrica,
   name: mydqmetrica
   incr_from: "2020-04-30 00:00:00",
   incr_to: "2020-05-01 00:00:00"
  }
 ]
}

```
пример: успех
```json
{
status: "SUCCESS",
intervals: []
}

```
