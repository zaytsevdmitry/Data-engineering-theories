# Статусная модель данных [датасетов](AbstractEntities/DataSet.MD)
Описывает именованным статусом инкрементальное состояние датасета.
Помимо именования интервал характеризуется моментом начала и моментом окончания. Эти значения присутствуют в качестве фильтра к источнику данных тк это одни и те же значения.

# Проблематика
## Границы инкремента не всегда равны интервалам запуска планировщика заданий.
На пример есть датасет который мы собираем раз в сутки, но запрос формирующий инкремент охватывает более глубокий период чем дата прошлого запуска.
Как правило, такая необходимость вызвана логикой изменения источника. На пример источник может получать изменения записей, которые созданы за пределом интервала, а отметка с указанием момента изменения не предоставляется.
## Интервалы запуска планировщика могут меняться.
На пример сначала нам нужно прогрузить историю крупными партиями по 1 году, а потом мы переходим в режим повседневного сбора суточными интервалами 

## Интервалы датасетов-источников могут не совпадать с интервалами собираемого датасета
На пример наш датасет собирается раз в сутки и охватывает записи измененные за пять дней, а на вход у него дав источника. Один собирается каждый час, другой раз в сутки.
Нужно обеспечить проверку отсутствия пропусков в сборе данных

## Датасеты меняющиеся онлайн и/или в потоке
По таким датасетам сложно отслеживать интервалы инкрементов

* ни одна потоковая система не предоставляет подобной функциональности тк считается, что это элемент логики данных.

* как и в случае с расписанием на стороне отправляющей системы требуется обеспечение датой изменения

Иначе говоря отправляющая система обязана обеспечить что-то вроде технологического поля с моментом изменения. Принимающая система обязана обработать само сообщение, поле с моментом изменения, сообщить в сервис статусную запись. Выполняется что-то вроде двухфазной фиксации.

Для случаев когда отправляется изменение класса crud важен порядок изменений, поэтому требуется еще одно служебное поле счетчик.

В асинхронной среде нужно обеспечить отправку ключа записи в ключ распределения
это также обязанность отправляющей системы

Счетчик и ключ распределения должны обеспечить на потребителе следование порядку изменений даже в распределенном режиме.

# Варианты использования 
* В механизме контроля в процессе сборки датасета - проверяются зависимость на предмет готовности их использования
* В DataQuality - отображение полноты и свежести датасета 
* В мониторинге - составление трешхолдов и нотификаций

# Атрибуты состояний
Очевидно инкременты характеризуются интервальностью и статусом
* name - ключевое уникальное имя на уровне системы. Это не обязательно одно поле. На пример если уникальность датасета обеспечивается на уровне пространства имен (namespace), то дополнительно потребуется и указатель на пространство к которому относится датасет.
* type - тип указывает на тип объекта по которому контролируется интервал. Возможные типы:
  * dataSet - указатель на датасет
  * dataqualityMetrica - указывает на метрику качества датасета

* intervalStart - временная метка от которой начинается интервал
* intervalStart - временная метка от которой начинается следующий интерва
* status - обозначение состояния в котором находится инкремент.
  Возможные статусы:
  * FAILED есть проблемы
  * SUCCESS все хорошо
  * INPROGRESS изменение производится в данный момент, для использования это то же самое что failed


Mодель может иметь следующий вид:

```
|name     |type   |intervalStart       |intervalStart      | status|
|---------|-------|--------------------|-------------------|-------|
|myDataSet|dataSet|2020-01-01 00:00:00z|2021-01-01 00:00:00|FAILED |
```
При этом не нужно учитывать периодичность 

# Контракты

## Изменение состояния 

```json
{
"dataSet": "myDataSet",
"status": "FAILED",
"type": "dataSet",
"incr_from": "2020-01-01 00:00:00",
"incr_to": "2021-01-01 00:00:00"
}
```

Сервис должен зафиксировать полученный интервал, и отправить уведомления об изменении всем подписчикам.
## Запрос состояния

```json
{
"dataSet": "myDataSet",
"incr_from": "2020-04-01 00:00:00",
"incr_to": "2020-05-01 00:00:00"
}

```
В ответ на такой запрос сервис производит поиск "отсутствующих данных" в указанном временном ограничении. Проверяются состояния не только самого датасета, но и связанных с ним проверок качества данных. Если в запрошенном интервале есть пробелы, то сервис возвращает их списком, чтобы клиент помимо отказа в операции, мог еще и вывести причину отказа, по которой можно начинать принимать меры. В случае успеха сервер отправляет статус, с пустым списком.



Пример: нехватает 1 суток

```json
{
"status": "FAILED",
"intervals": [
  {
   "type": "dataSet",
   "name": "myDataSet",
   "incr_from": "2020-04-30 00:00:00",
   "incr_to": "2020-05-01 00:00:00"
  },
  {
   "type": "dataqualityMetrica",
   "name": "mydqmetrica",
   "incr_from": "2020-04-30 00:00:00",
   "incr_to": "2020-05-01 00:00:00"
  }
 ]
}
```
Пример: успех

```json
{
"status": "SUCCESS",
"intervals": []
}
```
